§0. МЕСТО ДОКУМЕНТА В ИЕРАРХИИ АБСТРАКЦИЙ
Согласно Конституции v3.0, проект организован в четыре уровня:

I.   ONTOLOGY      — Что существует?           (Object, Knowledge, Relation...)
II.  CONSTITUTION  — Какие законы действуют?    (SSOT, Owner, Lifecycle...)
III. ARCHITECTURE  — Как законы воплощаются?    ← ЭТОТ ДОКУМЕНТ
IV.  TECHNOLOGY    — Какие инструменты?         (Git, Markdown, YAML...)

Этот документ описывает пространства и правила существования объектов в HONC Research Library. Он не описывает:

    Конкретные каталоги (это зона Technology)
    Форматы файлов (это зона Technology)
    Онтологические сущности (это зона Ontology)
    Законы библиотеки (это зона Constitution)

Принцип: Архитектура должна пережить смену GitHub на объектную БД, Markdown на XML, файловой системы на графовую модель. Если изменение технологии требует изменения этого документа — документ составлен неправильно.
§1. ТРИ ПРОСТРАНСТВА ПРОЕКТА
HONC Research Library организована в три онтологически различных пространства. Каждое пространство отвечает на свой вопрос и подчиняется своим правилам.
1.1. Workspace — Пространство происхождения знаний
Вопрос: «Почему библиотека выглядит именно так?»
Назначение: Хранит процесс рождения знаний — не только черновики, но и:

    Обсуждения (как provenance решений)
    Эксперименты и их результаты
    Тупиковые ветви и отвергнутые гипотезы
    Альтернативные решения
    Журналы решений (Decision Context)

Онтологический статус: Workspace — это пространство происхождения (provenance space). Объекты Workspace не являются частью канона, но объясняют, как канон был сформирован.
Правила:

    Объекты Workspace не требуют утверждения Автором
    Объекты Workspace могут противоречить друг другу (это фиксирует историю поиска)
    Переход из Workspace в Library требует прохождения Review → Approved

1.2. Library — Пространство канонического знания
Вопрос: «Что является установленным знанием?»
Назначение: Хранит утверждённые объекты, организованные по происхождению материала.
Структура (верхний уровень — по происхождению):

    Canon — утверждённое Автором знание
    Sources — первоисточники (книги, статьи, архивы)
    Research — рабочие материалы экспериментов
    Assets — нетекстовые ресурсы (языконезависимые)

Внутри Canon — тематическая структура (HONC / TFF / TD), отражающая трёхуровневую архитектуру теории.
Правила:

    Все объекты Library подчиняются законам Конституции (SSOT, Lifecycle, Provenance)
    Тематическая структура Canon — это классификация, а не структура хранения
    Тип объекта (аксиома, теорема, определение, рисунок) хранится в метаданных, а не в структуре

1.3. Editions — Пространство представлений
Вопрос: «Как знание показывается читателю?»
Назначение: Собирает представления знаний для конечного потребителя — книги, статьи, Wiki, переводы.
Принцип: Editions не содержат знаний. Они собирают знания из Library через ссылки. Изменение объекта Library автоматически требует обновления всех использующих его представлений.
Структура:

    Ru — Authoritative Formulation (сегодня)
    En — Derived Representation
    Nd — Derived Representation

Правила:

    Каждое представление ссылается на объекты Library, а не копирует их
    Языковая версия — это свойство представления, а не свойство знания
    Язык Authoritative Formulation определяется Автором и может меняться исторически

§2. ТИПЫ ОБЪЕКТОВ И ОРТОГОНАЛЬНЫЕ КЛАССИФИКАЦИИ
В архитектуре используются две ортогональные классификации, которые не должны смешиваться.
2.1. Классификация 1: Тема (где объект живёт в Canon)

    HONC — онтология, структура (что существует)
    TFF — формализм, математический аппарат (как описывается)
    TD — динамика, физика процессов (как проявляется)

Тема меняется редко (раз в годы).
2.2. Классификация 2: Тип объекта (метаданные)

    Axiom — исходное положение, не выводимое из других
    Definition — точное описание понятия
    Theorem — утверждение, выводимое из аксиом
    Figure — визуальное представление
    Dataset — набор данных
    Experiment — описание эксперимента
    Decision Context — зафиксированное решение (см. §3.3)
    Description — текстовое описание другого объекта (см. §3.2)

Тип объекта может добавляться без изменения структуры.
Принцип: Любое изменение дерева каталогов при появлении нового типа объекта — признак архитектурной ошибки.
§3. СПЕЦИАЛЬНЫЕ ТИПЫ ОБЪЕКТОВ
3.1. Description как отдельный объект
Рисунок (Asset) и его описание — это два разных объекта, связанных отношением describes:

Asset (рисунок)
    │
    described_by
    │
Description (текст)

Почему не в метаданных:

    Description может изменяться независимо от Asset
    Description может переводиться (языкозависимый объект)
    Description может иметь собственную историю версий
    Description может ссылаться другими объектами

Description обладает всеми признаками самостоятельного Object (§3.1 Конституции).
3.2. Decision Context вместо Discussion
Обсуждение — бесконечный поток сообщений, не имеющий естественной атомарности. Decision Context — атомарный объект решения:

Decision Context
├── Тема: [конкретный вопрос]
├── Участники: [роли]
├── Результат: [принятое решение]
└── Provenance: [ссылки на обсуждения, материалы]

Тысячи сообщений являются происхождением одного Decision Context.
3.3. Reference vs Dependency
Два семантически различных типа связи с внешними системами (например, лабораторией NHC-TD):

    Dependency — логическая зависимость (TD использует эксперимент NFSI-01; если эксперимент изменится, TD может потребовать пересмотра)
    Reference — информационная ссылка («см. лабораторный журнал»; не влияет на статус объекта)

Принцип: Смешивание Reference и Dependency создаёт неоднозначность в графе знаний.
§4. IDENTITY RULES — ПРАВИЛА ИДЕНТИЧНОСТИ ОБЪЕКТА
Критический раздел. Без этих правил SSOT разрушается — разные участники будут принимать разные решения о том, является ли изменение новой версией или новым объектом.
4.1. Идентичность сохраняется (новая версия)
Изменение не создаёт новый объект, если:

    Исправлена опечатка или форматирование
    Изменены метаданные (статус, дата, автор)
    Уточнены формулировки без изменения смысла
    Добавлены ссылки на другие объекты
    Добавлены новые представления (Representation) того же Object

4.2. Идентичность меняется (новый объект)
Изменение создаёт новый объект, если:

    Изменилось математическое содержание (формула, доказательство)
    Изменилась физическая интерпретация
    Добавлено или удалено отношение (Relation)
    Изменилась структура (для составных объектов)
    Объект переформулирован так, что старая и новая версии не могут одновременно быть истинными

4.3. Спорные случаи
Решение о сохранении идентичности в спорных случаях принимает владелец объекта (§4.2 Конституции) с консультацией Хранителя онтологии. Решение фиксируется в Decision Context.
§5. ОТНОШЕНИЯ МЕЖДУ ОБЪЕКТАМИ
5.1. Типы отношений
Отношение	Семантика
depends_on	A не может существовать без B
used_by	A используется в B
derives_from	A логически выводится из B
contradicts	A и B не могут быть одновременно истинны
extends	A расширяет B, не отменяя его
describes	A описывает B (для Description → Asset)
provenance_of	A является происхождением B (для Workspace → Library)
references	A ссылается на B без зависимости (см. §3.3)

Ссылка осуществляется через уникальный идентификатор объекта (ID), а не через путь в файловой системе. Это обеспечивает устойчивость архитектуры к реорганизации.
Конкретный формат ID (UUID, числовой, именованный) — это решение уровня Technology.
5.3. Запрет циклических ссылок
Отношения depends_on и derives_from не должны образовывать циклов. Отношения used_by и references могут образовывать циклы.
§6. РАЗМЕЩЕНИЕ ОБЪЕКТОВ В ПРОСТРАНСТВАХ
6.1. Что попадает в Workspace

    Черновики объектов (до стадии Review)
    Decision Context (журналы решений)
    Обсуждения (как provenance)
    Экспериментальные данные (до утверждения)
    Отвергнутые гипотезы
    Альтернативные решения

6.2. Что попадает в Library

    Canon: объекты со статусом Approved и выше
    Sources: первоисточники (книги, статьи, архивы)
    Research: утверждённые результаты экспериментов
    Assets: нетекстовые объекты (языконезависимые)

6.3. Что попадает в Editions

    Текстовые представления объектов Canon на разных языках
    Книги, статьи, Wiki, презентации, собранные из объектов Canon

6.4. Переходы между пространствами

Workspace ──(Review + Approval)──▶ Library ──(Assembly)──▶ Editions
     │                                  │
     └────────(Archive)─────────────────┘

Переход Workspace → Library требует прохождения полного жизненного цикла (§4.3 Конституции).
§7. ЯЗЫКОВАЯ АРХИТЕКТУРА
7.1. Язык как интерфейс, не измерение
Язык — это интерфейс доступа к знанию, а не его свойство. Знание (Knowledge) существует вне языка.
7.2. Языконезависимые объекты
Следующие типы объектов не имеют языка и хранятся в одном экземпляре в Library/Assets:

    CAD-модели
    SVG, PNG, HTML-анимации
    CSV-данные, датасеты
    Python-код
    STL, T-Flex файлы

7.3. Языкозависимые объекты
Следующие типы объектов имеют язык и существуют в виде Authoritative Formulation + Derived Representations:

    Книги, статьи
    Описания рисунков (Description)
    README, комментарии
    Глоссарии, подписи

7.4. Authoritative Formulation
Каждый объект библиотеки имеет одну Authoritative Formulation (§4.5 Конституции). Сегодня это русская версия. Через десятилетия это может быть английская. Конституция при этом не изменится.
§8. СВЯЗЬ С ЛАБОРАТОРИЕЙ NHC-TD
8.1. Разделение сущностей

    Лаборатория (NHC-TD) — пространство постоянного изменения (эксперименты, расчёты, обсуждения)
    Библиотека (HONC) — пространство стабильного хранения утверждённого знания

8.2. Логическая, не физическая связь
Библиотека содержит ссылки на лабораторию (через references), но лаборатория не является частью библиотеки.
Пример: Объект Library/Canon/TD/neutrino_oscillations может содержать ссылку references: ext:NHC-TD/experiments/NFSI-01/, но сам эксперимент физически хранится в репозитории лаборатории.
8.3. Перенос знания из лаборатории в библиотеку
Объект из лаборатории может перейти в Library/Canon только через полный жизненный цикл (Workspace → Review → Approved).
§9. КРИТЕРИИ ПРИЁМКИ АРХИТЕКТУРЫ
Архитектура считается корректной, если:

    ✅ Каждое решение выводится из законов Конституции
    ✅ Документ не содержит технологических деталей (Git, Markdown, YAML)
    ✅ Переход на другую технологию не требует изменения документа
    ✅ Все специальные типы объектов (Description, Decision Context) имеют онтологическое обоснование
    ✅ Identity Rules покрывают все типовые случаи
    ✅ Три пространства чётко разделены и не смешиваются
    ✅ Ортогональные классификации (тема vs тип) не смешиваются
