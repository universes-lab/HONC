📜 КОНСТИТУЦИЯ HONC RESEARCH LIBRARY v3.0 
Преамбула
HONC Research Library использует понятие "библиотека" как историческое название. По своей архитектуре она представляет собой систему управления научным знанием, основанную на принципах Single Source of Truth (SSOT) и строгого разделения уровней абстракции.
Настоящая Конституция определяет онтологические основы, законы и процессы, по которым существует эта система. Она не зависит от конкретных технологий хранения, форматов файлов или платформ публикации.
§1. УРОВНИ АБСТРАКЦИИ
Система HONC Research Library организована в четыре уровня абстракции. Каждый уровень отвечает на свой вопрос:
Уровень I: Ontology (Что существует?)
Определяет фундаментальные сущности, независимые от конкретного проекта:

    Object — минимальная неделимая сущность, обладающая уникальной идентичностью
    Knowledge — смысл, возникающий при интерпретации отношений между Object
    Relation — связь между Object (depends_on, used_by, derives_from, contradicts, extends)
    Representation — физическая форма, в которой существует Object (может быть несколько)
    Authoritative Formulation — первичное текстовое выражение Knowledge, созданное Автором

Уровень II: Constitution (Какие законы действуют?)
Определяет правила существования библиотеки:

    Закон единственности (SSOT) — каждый смысловой Object существует только один раз
    Закон владельца (Owner) — каждый Object имеет владельца, ответственного за его статус
    Закон жизненного цикла (Lifecycle) — каждый Object проходит через стадии: Idea → Draft → Review → Approved → Published → Deprecated → Archived
    Закон происхождения (Provenance) — каждый Object Канона обязан иметь происхождение (эксперимент, вывод, перевод, наблюдение, внешний источник, решение Автора)
    Закон авторитета (Authority) — каждое Knowledge имеет одну Authoritative Formulation; остальные языковые версии являются Производными представлениями
    Закон версионирования (Versioning) — при изменении Approved-Object создаётся новая версия, старая переходит в Deprecated

Уровень III: Architecture (Как эти законы воплощаются?)
Определяет структуру организации знаний:

    Canon — утверждённое Автором знание (структурируется по тематике: HONC/TFF/TD)
    Sources — первоисточники (книги, статьи, черновики, архивы)
    Research — рабочие материалы экспериментов
    Assets — нетекстовые ресурсы (рисунки, CAD, анимации, датасеты)
    Editions — языковые представления знаний (Ru/En/Nd)
    Workspace — среда разработки решений (обсуждения, черновики Конституции, ТЗ)

Уровень IV: Technology (Какие технологии используются?)
Определяет конкретные инструменты реализации:

    Git, GitHub
    Markdown, YAML
    CI/CD
    Obsidian, PDF, HTML, SVG
    T-Flex, Python

Принцип: Конституция (Уровень II) не должна зависеть от технологий (Уровень IV). Изменение технологий не требует изменения Конституции.
§2. ФУНДАМЕНТАЛЬНЫЙ ПРИНЦИП
Knowledge не хранится. Хранится только Object.
Knowledge появляется при интерпретации отношений между Object.
Из этого принципа автоматически выводятся все остальные законы:

    Почему книга не источник знания? Потому что книга — Representation, а не Object.
    Почему перевод не источник знания? Потому что Translation — Representation.
    Почему SVG и PNG не дублирование? Потому что это два Representation одного Object.
    Почему статья может устареть? Потому что она использует старые связи между Object.

§3. ОПРЕДЕЛЕНИЯ
§3.1. Object (Объект)
Object — это минимальная неделимая сущность HONC Research Library, обладающая:

    Уникальной идентичностью (ID)
    Метаданными (статус, версия, автор, дата)
    Отношениями (зависимости, использование, вывод)
    Происхождением (provenance)
    Нулём или более представлениями (representation)

Примеры объектов:

    Определение (например, "квантрон")
    Рисунок (например, структура Qν-латчера)
    CAD-модель (например, 3D-модель латчера)
    Датасет (например, SNO+ event-level данные)
    Git-коммит (например, изменение в коде)
    Обсуждение (например, протокол встречи команды)

§3.2. Representation (Представление)
Representation — это физическая форма, в которой существует Object. Один Object может иметь несколько представлений.
Примеры:

    Object "фотография" может иметь представления: PNG, SVG, TIFF, JPEG, WEBP
    Object "CAD-модель" может иметь представления: исходник, рендер, миниатюра, HTML-анимация, GIF, видео

§3.3. Relation (Отношение)
Relation — это связь между Object. Knowledge возникает из отношений между Object.
Типы отношений:

    depends_on — Object A зависит от Object B
    used_by — Object A используется в Object B
    derives_from — Object A выводится из Object B
    contradicts — Object A противоречит Object B
    extends — Object A расширяет Object B

§3.4. Provenance (Происхождение)
Provenance — это источник, из которого возник Object. Каждый Object Канона обязан иметь происхождение.
Типы происхождения:

    Эксперимент
    Вывод
    Перевод
    Наблюдение
    Внешний источник
    Решение Автора

§3.5. Authoritative Formulation (Авторитетная формулировка)
Authoritative Formulation — это первичное текстовое выражение Knowledge, созданное Автором.
Принцип: Язык Authoritative Formulation определяется Автором и может меняться исторически. Сегодня почти все Authoritative Formulation будут на русском языке. Через двадцать лет это может уже не быть так. Конституция при этом не изменится.
Все остальные языковые версии являются Производными представлениями (Derived Representations) и обязаны ссылаться на Authoritative Formulation.
§4. ЗАКОНЫ БИБЛИОТЕКИ
§4.1. Закон единственности (SSOT — Single Source of Truth)
Каждый смысловой Object существует в библиотеке только один раз.
Object может иметь несколько физических представлений (representation), но все они ссылаются на один смысловой Object.
Следствие: Запрещено дублирование текста между Object. Разрешены только ссылки на Object, но не их копии.
§4.2. Закон владельца (Owner)
Каждый Object имеет владельца.
Владелец — это роль, ответственная за:

    Создание Object
    Утверждение статуса Object
    Внесение изменений в Object
    Архивирование Object

Роли владельцев:

    Автор — владелец объектов Канона
    Редактор — владелец объектов Sources
    Хранитель — владелец объектов Research и Assets
    Исполнитель — владелец объектов Editions (только в части сборки)

§4.3. Закон жизненного цикла (Lifecycle)
Каждый Object проходит через следующие стадии:

Idea → Draft → Review → Approved → Published → Deprecated → Archived

Стадия	Описание	Доступ
Idea	Зародыш Object, не оформленный	Только владелец
Draft	Черновик, требует доработки	Владелец + редакторы
Review	Проходит экспертизу	Владелец + эксперты
Approved	Утверждён, готов к публикации	Все (чтение)
Published	Включён в представления	Все (чтение)
Deprecated	Устарел, заменён новой версией	Все (чтение, не цитируется)
Archived	Архивирован, не используется	Только администратор

Переходы между стадиями:

    Idea → Draft: решение владельца
    Draft → Review: решение владельца + согласие редактора
    Review → Approved: решение Автора после консультаций
    Approved → Published: решение Автора
    Published → Deprecated: создание новой версии (§4.6)
    Deprecated → Archived: решение Хранителя через 1 год после Deprecated

§4.4. Закон происхождения (Provenance)
Каждый Object Канона обязан иметь происхождение (provenance).
Object без происхождения не может быть включён в Канон.
§4.5. Закон авторитета (Authority)
Каждое Knowledge имеет одну Authoritative Formulation.
Остальные языковые версии являются Производными представлениями и обязаны ссылаться на Authoritative Formulation.
§4.6. Закон версионирования (Versioning)
Каждый Object имеет версию.
При изменении утверждённого Object:

    Создаётся новая версия Object
    Старая версия переходит в статус Deprecated
    Все ссылки на старую версию автоматически переносятся на новую (если это возможно)

Формат версии: v{major}.{minor} (например, v1.0, v1.1, v2.0)
§5. РОЛИ ПРОЕКТА
Конституция определяет роли, а не конкретных исполнителей. Соответствие ролей конкретным участникам проекта фиксируется в отдельном организационном документе.
Роль	Функция
Автор	Утверждение архитектуры, канонических документов, переходов между стадиями
Хранитель онтологии	Онтологическая чистота, проверка SSOT, соответствие Конституции
Методолог	Методологическая проверка, разделение уровней, экспертиза процессов
Предметный эксперт	Теоретическая экспертиза, проверка формул, физическая корректность
Архитектор реализации	Координация строительства, контроль качества, техническая архитектура
Исполнитель	Создание папок, перенос файлов, git, повседневная работа

§7. СТРУКТУРА БИБЛИОТЕКИ
§7.1. Верхний уровень (по происхождению)
Библиотека делится на шесть разделов верхнего уровня:

    Canon — утверждённое Автором знание
    Sources — первоисточники (книги, статьи, черновики, архивы)
    Research — рабочие материалы экспериментов
    Assets — нетекстовые ресурсы (рисунки, CAD, анимации, датасеты)
    Editions — языковые представления знаний
    Workspace — среда разработки решений

§7.2. Canon (Канон)
Canon — это множество Object, утверждённых Автором как истинное знание HONC.
Канон обладает свойствами:

    Единственность: каждое утверждение существует в Каноне только один раз (§4.1)
    Непротиворечивость: объекты Канона не противоречат друг другу
    Ссылочная целостность: все ссылки в Каноне ведут на существующие объекты
    Происхождение: каждый Object Канона имеет происхождение (§4.4)

§7.3. Второй уровень (внутри Canon)
Внутри Canon объекты организуются по тематике:

    HONC (Hierarchy of Nested Continua) — онтология, структура
    TFF (Temporal Fields Formalism) — математический аппарат, поле времени
    TD (Temporal Dynamics) — физика процессов, следствия

§7.4. Связь с лабораторией (NHC-TD)
Библиотека (HONC) и лаборатория (NHC-TD) — это разные сущности.

    Лаборатория — место, где проводятся эксперименты, расчёты, обсуждения. Она постоянно меняется.
    Библиотека — место, где хранятся утверждённые знания. Она стабильна.

Принцип связи: Библиотека содержит ссылки на лабораторию, но лаборатория не является частью библиотеки.
Пример: Object Canon/TD/neutrino_oscillations.md может содержать ссылку [Эксперимент NFSI-01](https://github.com/universes-lab/NHC-TD/experiments/NFSI-01/), но сам эксперимент хранится в репозитории лаборатории.
§8. ПРОЦЕССЫ
§8.1. Создание Object

    Владелец определяет тип Object и его границы (§3.1)
    Владелец создаёт Object в стадии Draft
    Владелец заполняет метаданные: ID, тип, статус, автор, дата, происхождение
    Владелец уведомляет редактора о создании Object

§8.2. Утверждение Object

    Владелец переводит Object в стадию Review
    Проводится экспертиза (минимум два эксперта)
    Эксперты дают рекомендации: утвердить, доработать, отклонить
    Автор принимает окончательное решение
    Если утверждён — Object переходит в стадию Approved

§8.3. Изменение Object

    Владелец создаёт новую версию Object (§4.6)
    Новая версия проходит процесс утверждения (§8.2)
    После утверждения новая версия переходит в Approved
    Старая версия переходит в Deprecated
    Все ссылки на старую версию автоматически переносятся на новую (если возможно)

§8.4. Публикация Object

    Автор решает, в какие представления включить Object
    Исполнитель собирает представление из объектов Канона
    Представление проходит проверку на ссылочную целостность (§4.1)
    Представление публикуется

§8.5. Архивирование Object

    Хранитель определяет, что Object устарел
    Object переводится в стадию Deprecated
    Через 1 год Object переводится в Archived
    Object удаляется из всех представлений
    Ссылки на Object заменяются на ссылки на новую версию (если есть) или удаляются

§9. ПРИНЦИП ЭКСПЕРТНЫХ КОНСУЛЬТАЦИЙ
Ни одно архитектурное решение не принимается без рассмотрения минимум двумя независимыми экспертами.
§9.1. Типы экспертизы
Область	Эксперт 1	Эксперт 2
Архитектура	Методолог	Хранитель онтологии
Физика	Предметный эксперт (внутренний)	Предметный эксперт (внешний)
Организация	Архитектор реализации	Методолог

§9.2. Роли экспертов

    Предметный эксперт (внутренний) — знает историю разработки и способен оценить соответствие новым идеям уже существующей логике HONC
    Предметный эксперт (внешний) — оценивает теорию так, как её увидит новый исследователь, без груза предыдущих обсуждений

Принцип: Сочетание внутренней и внешней экспертизы обеспечивает баланс между глубиной понимания и независимостью оценки.
§10. УРОВНИ ПРИНЯТИЯ РЕШЕНИЙ
§10.1. Уровень A — Конституционный
Что меняется: Архитектура библиотеки, законы SSOT, жизненный цикл объектов, структура HONC.
Кто принимает: Только Автор после консультаций.
Частота: Крайне редко (не чаще раза в год).
§10.2. Уровень B — Архитектурный
Что меняется: Структура каталогов, правила хранения, форматы документов, система ссылок.
Кто принимает: Архитекторы (Методолог + Хранитель онтологии) готовят решение, Автор утверждает.
Частота: По мере необходимости.
§10.3. Уровень C — Технологический
Что меняется: Git, CI, шаблоны, автоматизация, структура README.
Кто принимает: Архитектор реализации и Исполнитель. Архитекторы вмешиваются только при необходимости.
Частота: Регулярно.
§10.4. Уровень D — Рабочий
Что меняется: Перенос рисунков, исправление ссылок, перевод книги, каталогизация файлов, оформление Wiki.
Кто принимает: Исполнитель.
Частота: Ежедневно.
§11. ЗАКЛЮЧИТЕЛЬНЫЕ ПОЛОЖЕНИЯ
§11.1. Приоритет Конституции
В случае противоречия между Конституцией и любым другим документом библиотеки, приоритет имеет Конституция.
§11.2. Изменение Конституции
Изменение Конституции требует:

    Предложения от любого участника проекта
    Обсуждения с экспертами (§9)
    Утверждения Автором

§11.3. Вступление в силу
Конституция вступает в силу после утверждения Автором.
